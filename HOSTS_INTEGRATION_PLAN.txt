Minecraft P2P Connector: analysis of projects from G:\minecraftjava\hosts
Date: 2026-04-11

1. Current diagnosis of our app

- The current app already has three transport ideas:
  - direct libp2p/QUIC/DCUtR
  - relay fallback through our own relay module
  - a partial e4mc QUIC flow that can obtain a public domain
- The current e4mc integration is most likely incomplete. In our code, src-tauri/src/network/e4mc.rs opens the QUIC control stream, asks the broker/relay for a domain, then bridges inbound QUIC streams to local TCP Minecraft.
- The real e4mc project contains much more than "get domain + accept streams":
  - QuiclimeSession.java
  - Dialtone-related classes
  - ticket smuggling classes
  - address resolver / DNS TXT based redirect logic
  - multiple mixins around connection and server address handling
- Practical conclusion: the domain being shown in our UI does not prove that the whole e4mc path is protocol-correct for real players. Right now we likely implemented only the visible part of e4mc, not the full compatible behavior.

2. Per-project analysis

2.1 e4mc-minecraft-architectury-rererewrite

What it is:
- Real e4mc multi-loader client mod.
- Uses QUIC/Quiclime and additional Dialtone/Ticket plumbing.

Interesting parts:
- Broker-driven relay selection.
- QUIC relay transport with multi-stream model.
- Public domain UX: human-readable e4mc link.
- Version-agnostic tunnel idea: transport raw Minecraft traffic instead of parsing protocol payloads.
- Extra resolver/ticket machinery for making relay/domain behavior actually work.

What to borrow:
- The architecture and protocol understanding.
- Broker selection logic.
- Domain assignment flow.
- Health/doctor style diagnostics.

What not to blindly copy:
- Do not assume "domain assigned" means "playable".
- Do not partially port classes and call it e4mc-compatible.
- Do not tie app reliability to undocumented e4mc internals unless we either:
  - achieve real protocol parity, or
  - run our own service with our own compatibility guarantees.

Recommendation:
- Keep e4mc as a research source and as a possible optional public-link mode.
- Treat "full e4mc compatibility" as a dedicated track, not a side feature.

2.2 world-host-main

What it is:
- A mature Minecraft hosting mod with central coordination.
- Uses two major strategies:
  - UPnP / temporary public port if available
  - proxy mode if direct path is not possible

Interesting parts:
- Very useful UDP hole punch packet flow document.
- Port lookup flow is simple and realistic.
- "tempip" concept is good UX: a temporary public endpoint that stays alive once peers connect.
- Direct/proxy fallback is a solid product pattern.

What to borrow:
- Port lookup + UDP signaling ideas.
- UPnP-first, proxy-second decision model.
- Temporary public endpoint UX.

What not to copy directly:
- Friends system and mod-specific UX are not relevant to our desktop app.
- Their server/protocol is designed around their own ecosystem.

Recommendation:
- Borrow the direct/proxy fallback philosophy and the UDP lookup flow.
- Add UPnP/NAT-PMP as an early host optimization before paying relay cost.

2.3 stealthpipe-mod-main

What it is:
- Client-side mod focused on hostile networks.
- Uses WebRTC when possible, then falls back to WSS relay over TCP/443.

Interesting parts:
- Strongest idea for "absolute majority of players with bad NAT / hostile Wi-Fi":
  - if UDP/direct fails, tunnel through TLS WebSocket on 443
- Parallel per-player connections reduce head-of-line blocking.
- Packet batching lowers PPS on the relay.
- Product goal is exactly the problem we need to solve: networks where QUIC or custom UDP paths are unreliable or blocked.

What to borrow:
- 443/WSS relay fallback as the "last-resort but reliable" path.
- Parallel data channels per player.
- Explicit versioning / protocol version checks.
- Packet batching as an optional optimization.

What not to copy directly:
- Their mod hooks directly into Netty/Minecraft internals.
- We should not build our desktop app around Minecraft mixins.

Recommendation:
- This is the best source for the relay fallback design of our app.
- We should replicate the transport concept in the desktop app, not the mod injection layer.

2.4 stealthpipe-relay-main

What it is:
- Relay-side implementation for StealthPipe.
- Stateless Go service with signaling + room creation + traffic forwarding.

Interesting parts:
- Very strong candidate for our fallback relay service.
- Cheap to host, container-friendly.
- Has PoW and anti-abuse primitives.
- Uses standard HTTPS/WSS hosting model.
- Designed for bad networks, not just ideal NAT cases.

What to borrow:
- Overall relay topology.
- Room creation + signal/data separation.
- Anti-abuse approach: PoW, bandwidth controls, stateless rooms.
- Hosting model.

What to change for our app:
- Replace mod-specific room/player protocol with desktop-app session protocol.
- Keep data opaque and version-agnostic.
- Integrate with our own presence/session metadata.

Recommendation:
- Highest-value project in the folder for real reliability gains.
- If we want "works for almost everyone", this should be the core of the hard fallback path.

2.5 p2ptunnel-master

What it is:
- General-purpose libp2p/noise tunnel tool for TCP/UDP.
- Supports direct or relayed encrypted tunnels.

Interesting parts:
- Good generic reference for building encrypted tunnels.
- Useful as a conceptual comparison point for direct + relay tunneling.

Weakness for our use case:
- Not Minecraft-specific UX.
- No polished public-link flow.
- Less directly useful than StealthPipe/world-host for player-facing reliability.

Recommendation:
- Use only as a protocol/reference source.
- Do not integrate it directly into the app.

2.6 P2P-1.20.4

What it is:
- Fabric mod that appears to use an external helper binary (GOLE) and local tunnel setup.

Interesting parts:
- Demonstrates external helper process approach.

Weakness for our use case:
- Depends on external process management.
- More brittle operationally.
- Tied to specific mod flow and version range.
- Much weaker long-term fit than native desktop-app transports.

Recommendation:
- Do not build the new core around this project.
- At most, use it as a cautionary example of what we do not want: opaque external binary dependency.

3. What is most useful for our application

Priority order:

1) stealthpipe-relay-main
- Best basis for a reliable "works on harsh NAT / blocked UDP / public Wi-Fi" path.

2) stealthpipe-mod-main
- Best transport strategy reference for 443/WSS fallback and multi-connection relay behavior.

3) world-host-main
- Best source for port lookup, UDP signaling flow, and direct/proxy product logic.

4) e4mc-minecraft-architectury-rererewrite
- Best source for public domain UX and QUIC relay concepts, but only worth using if we implement it properly end-to-end.

5) p2ptunnel-master
- Useful background reference, not a product integration target.

6) P2P-1.20.4
- Lowest value for core integration.

4. Recommended future network architecture for Minecraft P2P Connector

Goal:
- One desktop app on host.
- One desktop app on joiner.
- No dependence on Minecraft mod loader.
- Transport remains opaque to Minecraft payloads so it works across Minecraft versions.

Transport stack:

Layer A: Direct path
- Keep current libp2p direct path.
- Improve NAT diagnostics, retry policy, and hole punching telemetry.
- Add UPnP/NAT-PMP attempt before fallback.

Layer B: UDP-assisted public path
- Borrow world-host style port lookup and hole-punch orchestration.
- If temporary direct public mapping works, expose temporary join address.

Layer C: Hard fallback relay path
- Add StealthPipe-style WSS relay over 443.
- This should be the guaranteed fallback for most bad NAT / firewall environments.
- Relay traffic stays opaque and version-agnostic.

Layer D: Public URL mode
- Keep e4mc-style public link UX, but only when the backing transport is truly alive.
- "Show link only after end-to-end acceptance check succeeds" must be enforced.

5. Specific e4mc conclusion

Short version:
- Our current e4mc implementation should be treated as experimental / incomplete.

Why:
- The real e4mc project includes more than QUIC control and stream forwarding.
- It has extra resolver, ticket, and connection glue that we do not currently mirror.
- The user already observed the real symptom of incomplete integration:
  - domain generated
  - players without the mod could not actually join

What to do next:
- Either:
  - finish full e4mc-compatible behavior and validate it against real clients, or
  - stop calling the current mode "ready" and replace it with our own public-link flow backed by a transport we fully control.

Recommendation:
- For product reliability, I recommend using our own public-link mode on top of a transport stack we own:
  - direct libp2p
  - UDP-assisted mapping
  - WSS relay fallback
- e4mc-inspired UX is worth keeping.
- e4mc protocol parity is optional, not mandatory, unless interoperability with the official e4mc ecosystem is a hard product requirement.

6. Concrete integration plan

Phase 1: Fix the false-positive public-link problem
- Do not display "ready" public link until a transport health check passes.
- Add relay-side / path-side acceptance test before surfacing the domain.
- If e4mc path fails health checks, automatically downgrade to our internal fallback and log the reason.

Phase 2: Add strong fallback relay
- Build or adapt a StealthPipe-inspired relay service for desktop-app sessions.
- Use WSS on 443 for signal and/or data fallback.
- Make session protocol app-owned, not mod-owned.
- Add per-peer channels and strong observability.

Phase 3: Improve direct success rate
- Add UPnP/NAT-PMP.
- Add world-host-style UDP port lookup assist.
- Add better NAT classification in logs and UI.

Phase 4: Public link productization
- Domain/link should point only to a verified working route.
- Add "temporary link" and "persistent link" states.
- Add server-region / relay-region selection.

Phase 5: Cross-version guarantee
- Keep transport payload-agnostic.
- Avoid Minecraft protocol parsing in the app.
- Local app forwards raw TCP/UDP to Minecraft.
- This is the only sane way to support many Minecraft versions and launchers.

7. Recommended implementation decision

If the target is:
"100% work for the absolute majority of players, including hard NAT"

Then the best path is:

- Keep current direct libp2p path
- Add world-host-style mapping assistance
- Add a StealthPipe-style WSS relay on 443 as the hard fallback
- Keep e4mc-like public links only as a presentation layer above a verified working transport

In one sentence:
- The highest-value thing to integrate from hosts is not "copy e4mc link generation", but "copy StealthPipe-grade relay reliability and World Host-grade fallback orchestration", then expose it with a public-link UX.
